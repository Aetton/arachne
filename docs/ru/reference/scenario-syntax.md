# Синтаксис сценариев

Эта страница — канонический справочник по YAML-сценариям и будущий источник встроенной
справки редактора.

## Правила доступа

```yaml
access:
  view:
    match: all
    roles: [developer]
    teams: [backend]
  run:
    match: all
    roles: [developer, release-engineer]
    teams: [backend]
```

При `match: all` должны совпасть и роль, и команда. При `match: any` достаточно
совпадения по одной из осей. На администраторов ACL сценария не распространяется.

## Разделы описания

Описание сценария содержит:

- параметры формы;
- ручные, запланированные или цепочечные триггеры;
- упорядоченные шаги;
- имя паука для каждой запускаемой операции;
- входные параметры, зависящие от конкретного паука.

## Ссылки и подстановки

Шаг может ссылаться на параметры сценария и основной артефакт уже завершённого шага:

```yaml
with:
  version: "${params.version}"
  artifact: "${build.artifact}"
  package_url: "${build.download_url}"
  label: "broker-${params.version}"
```

Поддерживаются формы `${params.<name>}` и `${<step-id>.<artifact-field>}`. Полями
артефакта могут быть его стандартные атрибуты и ключи из метаданных артефакта.

Ссылки разрешаются рекурсивно во вложенных отображениях и списках любой глубины:

```yaml
with:
  params:
    targets:
      - name: broker
        packages:
          - version: "${params.version}"
            source: "${build.download_url}"
```

Если строка состоит только из одной ссылки, тип подставленного значения сохраняется.
Так можно передавать логические значения, числа, `null` и целые объекты артефактов
без преобразования в строку. Ссылка внутри более длинной строки преобразуется в текст.

Неизвестный параметр, неизвестный ID шага, некорректная ссылка или неподдерживаемая
форма ссылки приводят к ошибке шага, а не молча заменяются пустым значением. Сейчас
ссылка содержит ровно два сегмента пути. Произвольные выражения и пути наподобие
`${params.build.version}` не поддерживаются.

## Якоря и псевдонимы YAML

Arachne разбирает YAML через PyYAML `safe_load`, поэтому во входном YAML поддерживаются
обычные якоря, псевдонимы и ключи слияния:

```yaml
x-broker-params: &broker-params
  version: "${params.version}"
  release: "${params.release}"
  branch: "${params.branch}"

steps:
  - id: build-auth
    spider: scenario
    action: run
    with:
      scenario: build-broker-auth
      params: *broker-params

  - id: build-gateway
    spider: scenario
    action: run
    with:
      scenario: build-broker-gateway
      params:
        <<: *broker-params
        upload: true
```

Якоря относятся к синтаксису YAML-документа, а не к хранимой модели сценария.
При разборе они разворачиваются в обычные отображения и списки. Затем описание
сохраняется в базе как JSON-совместимое значение, поэтому после повторного открытия
или экспорта имена якорей и связи псевдонимов не сохраняются. Развёрнутые значения
останутся правильными, но после прохода через редактор или экспорт YAML может содержать
повторяющиеся блоки вместо якорей.

Используйте якоря как удобство при импорте, когда потеря исходного форматирования
допустима. Для сохранения якорей после редактирования пришлось бы хранить исходный YAML
либо использовать round-trip-парсер и модель хранения на основе YAML. Эту задачу не
следует смешивать с разрешением ссылок во время выполнения.

## Запуск другого сценария

Используйте паука `scenario`, когда один сценарий должен запустить другой в качестве
одного из упорядоченных шагов. В отличие от триггера `chain`, оркестрация остаётся
внутри родительского запуска: параметры передаются явно, дочерние логи транслируются,
родитель ждёт завершения, а итоговое состояние дочернего запуска передаётся
родительскому шагу.

### Минимальная форма

```yaml
steps:
  - id: build-auth
    spider: scenario
    action: run
    with:
      scenario: build-broker-auth
```

Поддерживаемое действие — `run`. Другие имена действий не входят в контракт паука
`scenario` и использоваться не должны.

### Входные параметры

`with.scenario`
: Обязательное поле. Точный slug опубликованного дочернего сценария. Значение должно
  быть непустой строкой. Неизвестный или недоступный slug приведёт к ошибке родительского
  шага при попытке запуска дочернего сценария.

`with.params`
: Необязательное отображение, передаваемое дочернему сценарию. Если поле не задано,
  дочерний сценарий получает пустое отображение параметров. До отправки значения
  разрешаются стандартным механизмом `${...}`, включая вложенные отображения и списки.

Пример с явной передачей параметров:

```yaml
steps:
  - id: build-auth
    spider: scenario
    action: run
    with:
      scenario: build-broker-auth
      params:
        version: "${params.version}"
        release: "${params.release}"
        branch: "${params.branch}"
```

`with.params` должен быть отображением. Списки, строки, числа и другие скалярные
значения отклоняются.

Параметры не передаются автоматически. Каждое значение, необходимое дочернему сценарию,
нужно явно объявить в `with.params`:

```yaml
# Параметр родителя "version" передаётся дочернему сценарию.
params:
  version: "${params.version}"
```

### Выполнение и логи

Дочерний сценарий создаётся как обычный сохраняемый запуск Arachne со своим ID запуска,
снимком сценария, записью истории, логами, состоянием и артефактами. Родительский шаг
ждёт, пока дочерний запуск достигнет конечного состояния.

Записи дочерних логов копируются в родительский шаг с префиксом из идентификаторов
дочернего сценария и шага:

```text
[build-broker-auth/checkout] cloning repository
[build-broker-auth/build] compilation complete
```

Так полная оркестрация видна из родительского запуска, а дочерний при этом остаётся
самостоятельным запуском, который можно изучить отдельно.

### Передача состояния

Конечное состояние дочернего запуска становится состоянием родительского шага
`scenario`:

- `success` — родитель переходит к следующему шагу;
- `failed` — родительский шаг завершается ошибкой, обычное последовательное выполнение останавливается;
- `cancelled` — родительский шаг отмечается отменённым, обычное последовательное выполнение останавливается.

Поскольку обычные шаги сценария выполняются последовательно, список шагов `scenario`
запускает дочерние сценарии один за другим. Следующий стартует только после того, как
предыдущий вернул `success`.

```yaml
steps:
  - id: build-auth
    spider: scenario
    action: run
    with:
      scenario: build-broker-auth

  - id: build-gateway
    spider: scenario
    action: run
    with:
      scenario: build-broker-gateway
```

### Артефакты

После завершения дочернего сценария его артефакты становятся артефактами родительского
шага. В метаданные каждого скопированного артефакта добавляется происхождение:

- `scenario` — slug дочернего сценария;
- `child_run_id` — сохранённый ID дочернего запуска.

Имя, тип и URL загрузки артефакта сохраняются, если они доступны.

### Паук `scenario` и триггер `chain`

Используйте `scenario`, когда оркестрацией владеет родитель:

- дочерний сценарий является явным упорядоченным шагом;
- параметры передаются явно;
- родитель ждёт дочерний сценарий;
- логи и артефакты видны в родительском запуске;
- ошибка дочернего сценария влияет на родителя.

Используйте `chain`, когда завершённый запуск должен независимо запустить другой сценарий:

- нижестоящий запуск управляется событием;
- он не является упорядоченным шагом вышестоящего сценария;
- вышестоящий запуск не ждёт его;
- результат нижестоящего запуска не меняет уже завершённый вышестоящий запуск.

### Текущие ограничения

Текущая реализация имеет следующие ограничения:

- отмена родительского запуска пока не распространяется на уже выполняющийся дочерний сценарий;
- рекурсивная оркестрация пока не защищена, поэтому сценарий может прямо или косвенно вызвать сам себя; авторы должны избегать циклов;
- выполнение дочернего сценария и передача живых логов пока требуют доступности дочернего запуска через тот же внутрипроцессный движок;
- права проверяются, когда пользователь запускает родительский сценарий, но внутренний запуск дочернего пока не выполняет отдельную проверку его ACL;
- личность вызывающего пока не передаётся в дочерний запуск, поэтому аудит владельца внутренних запусков требует доработки.

Это ограничения среды выполнения, а не возможности DSL. До появления явной отмены
дочерних запусков сценарии должны оставаться ацикличными и не зависеть от распространения
отмены от родителя.
